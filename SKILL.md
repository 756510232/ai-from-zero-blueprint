# 从零搭建领域陌生型 AI 应用：蓝图方法论

## 何时用本技能
- 用户想做一个具体领域的应用，但**用户和你（AI）对该领域都不熟**（世界杯预测、高考志愿、个人记账、简历优化、合同审查、学习卡片、食谱规划……）。
- 任务本质是"AI 可帮着实现"的，受众广、非小众。
- 用户**没有预设终点**——他只知道"想做个 XX 应用"，具体产品形态 / 受众 / 成功标准需要一起厘清。
- 用户问："怎么跟 AI 一步步沟通把这个做出来""该问 AI 什么问题""一共多少步""怎么验证对不对""完全不懂领域怎么靠 AI 做出优秀项目"。

## 核心信条：计算与解释分离（Compute / Explain）
任何成功的领域 AI 应用，其"真正干活的数字 / 排名 / 抽取"来自**确定性内核或检索内核**，**绝不来自 LLM 的自由生成**。LLM 只做一件事：把内核的客观输出翻译成人话。
- 内核错了 → 应用错了，但可定位、可测、可复现。
- LLM 错了 → 只是措辞难听，内核数字不变。

**一句话给用户的判断标准**：如果把这个应用里的 LLM 整个拔掉，剩下的部分还能不能独立算出结果？不能 → 架构有根本问题。

## 第零步：用 brainstorming 确认「你到底想要什么」（关键，勿跳过）

用户给的永远是**产品意图**（如"做一个用 AI 给高考生和家长做志愿建议的网页应用，张雪峰式直白表达"），**不是**"复盘 / 复现某个系统"之类的元任务。AI **不得假设终点**。

动手前，**先加载 superpowers 的 `brainstorming` skill**（用法见 `references/prompt-bank.md` 第零步），一次一问、多选优先，探索 2-3 个方案后分段呈现设计并逐段确认，最终把"终点"写进 `docs/plans/YYYY-MM-DD-<topic>-design.md`。

> **轻量化提示**：若用户**已给出具体产品意图**（如"做一个张雪峰式志愿网页应用"，而非"复盘/复现某系统"元任务），brainstorming **轻量化**——确认 + 锁定终点即可，**不要重做 2-3 方案探索**（避免把已明确的意图又推回"想当然"）。

brainstorming 必须澄清并锁定的「终点」：
- **产品形态**：网页 / CLI / API / 桌面端
- **目标受众**：谁用、解决他什么痛点
- **成功标准**：可衡量的验收口径（不是"好用"这种空话）
- **硬约束**：技术栈 / 合规 / 预算 / 数据可得性
- **明确不做什么**（YAGNI，ruthless 删减）

> ⚠️ 这一条直接纠正一个常见错误：把"复盘 / 复现某个已有系统"当成目标。本 Skill 的终点
> 是**用这套方法从零做出一个优秀的新项目**，参考仓库只是"优秀范式样例"，不是"要抄的对象"。
> "复盘"在本 Skill 里 = 拿一个真实目标完整走一遍方法论做出项目，而非复刻别人的代码。

## A 侦察（主动套用固化模板，不啰嗦反问）

终点确认后，AI **主动**套 `references/blueprint-template.md` 的通用模板产出以下内容（不再每次让用户问"三层是什么 / 有哪些陷阱"——这些是通用的，直接套）：

1. **概念表**：8-12 个领域术语 → 人话类比 + 在系统里的角色。
2. **三层落点**（套模板 §1）：写明每层负责 / 不负责；AI 明确"内核零 LLM import"这条铁律自检。
3. **陷阱**（套模板 §2 的 6 类 T1–T6）：给本项目具体实例 + 防法。
4. **数据需求**（套模板 §3）：标形态 / 可得性 / 降级映射。
5. **PROVE**：每个关键说法给出"为什么可信"（出处 / 成熟做法 / 已知数据）。

**原则**：侦察是 AI 主动完成的领域分析，不是向用户求证已知信息。用户已通过 brainstorming 给了意图，剩下 AI 当分析师去逼出内核。

---

## B 拆解（套标准 phase 顺序，聚焦「每步四要素」）

AI **主动套用**模板 §4 的固定 6 个 phase（P1 Schema → P2 最小内核 → P3 接真实数据 → P4 接解释层 → P5 健壮性降级 → P6 验证声明），填充本项目内容。**不必让用户重新"定义模块"——顺序通用。**

拆解的**核心交付物**是给每个 phase 写清**四要素**，用户看它来懂"每步边界与验收口径"：

| 要素 | 含义 |
|---|---|
| **约束目标** | 这一步要达成什么、不能做什么（如"禁止用未来数据""内核不得 import LLM"） |
| **完成标准** | 客观可检查（跑通 X 样本 / 数值落 Y 区间 / 测试通过） |
| **风险** | 易错点（过拟合 / 泄漏 / 外部漂移 / 数据难拿） |
| **检测方法** | 怎么客观验证对了（真实样本复核 / 手算对照 / 单测 / 对比已知答案） |

- 标注高风险 phase + 失败分支。
- 自然得出**总步数** = 固定模板的实例化（用户最关心的"一共几步"由此而来，不用临时发明）。

---

## C 建造（逐 phase，照四要素检测）

- **一次只做一个 phase**，对照其四要素验收，不过关不进下一 phase。
- 内核文件 `grep` 零 LLM / AI import（铁律自检）。
- 不偷偷做后续 phase；不一次性甩整套。
- 收尾逼 AI 列**诚实声明**：已验证 / 只是推测 / 用了降级 / 复现所需（种子 / 数据版本 / 依赖）。

---

## 四条铁律（任何领域都不能破）
1. **诚实降级**：缺数据 / 脏数据时不编造，明确标注"用代理值 / 未验证 / 此处降级"。
2. **全链路溯源**：每个对外数字都能追到来源（赔率行、向量块、条款、规则文件）。
3. **验证点显性**：每个 phase 有客观可查的验收标准（不是"看起来对"）。
4. **缺数据不编造**：没有就是没有。用占位 / 代理 / 降级，但绝不生成源里没有的数字。

## 阶段闸门（Phase Gate）模板
> ⚠️ `grep` 零 LLM import 是**必要非充分**自检：它只防"LLM 混进内核"，防不了逻辑错（如量纲方向反转）。必须配 P2 锁方向的单测才闭环。
每个 phase 结束前，必须能回答：
- [ ] 输出是什么（具体文件 / 数值）？
- [ ] 怎么客观验证它是对的？（测试 / 样本 / 对比）
- [ ] 缺 / 脏数据怎么处理的？有没有编造？
- [ ] 这一步的输出，下一步能否溯源到它？
不过关 → 不进下一 phase。

## 贯穿 5 问（每个 phase 都过一遍）
1. 如何**客观验证**这一步的结果？
2. **缺 / 脏数据**怎么办？（禁编造，要降级标注）
3. 数字能否**溯源**到来源？
4. 是否**过拟合 / 泄漏**（用未来数据训、全历史拟合称效果好）？
5. 交付里哪些**近似 / 降级 / 未验证**？

## 反模式红旗（出现即叫停）
- 🚩 AI 直接吐结论，没有确定性内核
- 🚩 单脚本全干（解析 + 算 + 解释混一起）
- 🚩 拿全历史拟合称"效果好"
- 🚩 缺数据就编个值填上
- 🚩 报告里出现源里没有的新数字
- 🚩 一次性甩整套架构试图"一步到位"
- 🚩 没有缺数据降级路径
- 🚩 随机模拟不固定种子、不可复现
- 🚩 LLM 既算又解释（角色没分开）
- 🚩 降级行被算成正常结论（伪装成推荐）——降级必须是独立显式状态
- 🚩 跨文件字段名漂移（无单一共享 schema 契约，loader 与数据各写各的）
- 🚩 硬约束写成空 `pass` 静默放过（资格 / 选科等必须显式 exclude）

| 7 | ai-investment-advisor | 投委会 | AKShare 取数+共识规则 | 多模型分析员 | GOOD |
| 8 | LossWeightEasily | 营养 | Mifflin-St Jeor BMR/TDEE+查表 | 可选食谱规划 | GOOD |

| 11 | dlai-roadmap | 学习路线 | 课程数据集+映射规则(零 LLM) | 无 | GOOD |
| 12 | FitMate | 健身 | 仅渐进超负荷+排期 | **LLM 抢动作选择** | WEAK |
| 13 | CookHero | 食谱生成 | 仅事后 sum，无求解器 | **LLM 自由编菜** | WEAK |
| 14 | ai-travel-planner | 旅行 | 仅算天数 | **LLM 包揽排程** | WEAK |
| 15 | AI-study-planner | 学习计划 | 仅算剩余天数 | **LLM 包揽排程** | WEAK |
| 16 | AI_Portfolio_Analyzer | 投资 | 波动率/β(确定) | **解释层空壳**(import 不调用) | WEAK |
| 17 | BabelDuck | 口语 | 状态/存储/路由外壳好 | **纠错纯 LLM 现编** | WEAK |

**结论**：17 个项目骨架同构，但只有 11 个真正落地了"确定性内核"；**6 个是薄壳**（LLM 抢硬计算或解释层空壳）。这 6 个恰恰是"外行靠 AI 容易写出来、但一碰就碎"的典型——本 Skill 的价值就是填这条沟：既教 GOOD 版怎么抄，也教 WEAK 版怎么纠正成 GOOD。

**源码带来的 6 个关键洞见**：
1. **确定性内核 ≠ 必须复杂 ML**：加权公式、纯算术、规则表、时间分块、数据集映射、查表——越轻越好，别一上来就上向量库。
2. **内核可委托给成熟外部系统**（green-deck→Anki；activity-timecard 甚至零 AI）。
3. **生成类护栏 = 源校验流水线**（green-deck 的 `src` 映回原文）。
4. **"计算/解释真分离"最简自检 = 内核文件 `grep` 不到任何 LLM import**。FitMate/CookHero/ai-travel-planner/AI-study-planner 都过不了这条。
5. **解释层缺失比内核错更隐蔽**（AI_Portfolio_Analyzer 空壳）——验收必须把"解释层真接上"列为 Gate。
6. **17 个里 6 个薄壳**：证明"外行靠 AI 写出来"与"写出 robust 项目"之间有方法论鸿沟，本 Skill 即填沟之用。

## 反模式纠正流程：把薄壳改造成有内核的项目
遇到 WEAK/ANTI 类项目（或你自己/AI 写出了"LLM 既算又解释"的版本），按此流程纠正：
1. **定位被 LLM 吃掉的硬计算**：问"这个成品里，哪些数字/决定是 LLM 直接编的？"
2. **逼出确定性替代**：和 AI 一起把那部分重写成"规则/公式/查表/算法"——
   - 匹配类 → 加权关键词/规则表（Resume-Matcher、activity-timecard）
   - 排程类 → 时间分块算法（study-planner）
   - 营养/财务类 → 公开公式 + 查表（LossWeightEasily、maybe）
   - 约束类 → 求解器（CookHero 应有）
3. **把 LLM 降级为解释层**：原 LLM 调用改为"消费已算好的内核结果，只生成文案/润色"。
4. **补解释层防空壳**：若原版解释层是空壳（如 AI_Portfolio_Analyzer），把内核指标拼成结构化输入喂给 LLM 写报告，列 Gate 验收"真接上"。
5. **自检**：内核文件 `grep` 无 LLM import；核心数字可手算复核；LLM 失败有降级。

## 实战演练（不是"复现"，是"从零做"）
本 Skill 的终点是**用方法论从零做出优秀项目**，不是复刻别人的代码。参考 `references/walkthrough.md`：
- 它拿一个**真实目标**（"张雪峰式高考志愿网页应用"）完整走一遍：brainstorming 确认终点 → A 主动侦察（套三层+陷阱+数据）→ B 拆解（标准 6 phase + 四要素）→ C 建造（真正写出冲稳保判定 SQL + 路由 + 解释层）。
- 17 个仓库只是"优秀范式样例"，用来对照"我的内核落点对不对"，而非抄。
- 每完成一个真实项目，即掌握"任意陌生领域都能照此法落地"的能力。

## 并行多路线压力测试（高风险 / 不确定领域强烈建议）

当领域陌生、或"怎么搭才不会碎"没把握时，**不要只跑一条路线**——让 2-3 个执行体并行用**不同技术路线**各做一遍同一目标，再汇总各路线**特有**的失败模式，合流成**一次** Skill 修订。这是把"17 个项目验证出的好骨架"真正落成"外行也不会写碎"的关键一跃。

- **路线示例**（同一目标，三套骨架）：
  - **A 纯确定性内核**：最小依赖、零 LLM，先把核心算对。
  - **B RAG + LLM 解释层**：检索路由 + 内核 + LLM 翻译，考验"解释层不越权 / 检索不漂移"。
  - **C schema 先行 + 诚实降级**：先签数据+输出契约，降级态当一等公民，考验"契约不漏 / 降级不伪装"。
- **执行约定**：每个执行体都写 `chat_log.md`（逐 phase 真实问题）；主 Agent 收齐后做**跨路线逐 phase 对照表**，定位方法论缺口（而非只改某一路线）。
- **合流**：把重复踩的坑（如 A、B 都踩的"方向反转""注释吞表头"）提炼成**强制护栏**（单测锁方向、loader 断言表头===契约），写回 `blueprint-template.md` / `SKILL.md`，而不是又一次散文提醒。
- **价值**：单路线只能证明"这条路能跑"；多路线并行才暴露"哪类坑是方法论没拦住的"——这正是本 Skill 持续变强的飞轮。

## 交付清单（给用户的最终产物）
建造完成后，至少交付：
1. 三层骨架图（你的项目落在哪层、内核算什么）
2. phase 列表 + 各自四要素 + Gate 状态
3. 验证点说明（怎么证明对）
4. 诚实声明（降级 / 未验证 / 推测项）
5. 复现方式（种子 / 数据版本 / 依赖）
6. **构建日志（chat_log）**：逐 phase 记录真实遇到的问题与修复——"看起来对"的坑只有写下来才暴露，它是反哺本 Skill 的原材料
