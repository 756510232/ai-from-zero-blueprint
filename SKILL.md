# 从零搭建领域陌生型 AI 应用：蓝图方法论

## 何时用本技能
- 用户想做一个具体领域的应用，但**用户和你（AI）对该领域都不熟**（世界杯预测、高考志愿、个人记账、简历优化、合同审查、学习卡片、食谱规划……）。
- 任务本质是"AI 可帮着实现"的，受众广、非小众。
- 用户问："怎么跟 AI 一步步沟通把这个做出来""该问 AI 什么问题""一共多少步""怎么验证对不对""完全不懂领域怎么靠 AI 理清"。

## 核心信条：计算与解释分离（Compute / Explain）
任何成功的领域 AI 应用，其"真正干活的数字 / 排名 / 抽取"来自**确定性内核或检索内核**，**绝不来自 LLM 的自由生成**。LLM 只做一件事：把内核的客观输出翻译成人话。
- 内核错了 → 应用错了，但可定位、可测、可复现。
- LLM 错了 → 只是措辞难听，内核数字不变。

**一句话给用户的判断标准**：如果把这个应用里的 LLM 整个拔掉，剩下的部分还能不能独立算出结果？不能 → 架构有根本问题。

## 三层同构骨架（跨 7 个领域通用）
```
① 确定性 / 检索内核  → 产生客观结果（概率 / 排名 / 抽取 / 分数 / 调度）
② 调度 / 路由层      → 决定调哪个内核、何时调、如何组合
③ AI 解释层          → 只把内核结果翻译成自然语言，绝不参与计算
```
外裹四条铁律（见下）。

## 四条铁律（任何领域都不能破）
1. **诚实降级**：缺数据 / 脏数据时不编造，明确标注"用代理值 / 未验证 / 此处降级"。
2. **全链路溯源**：每个对外数字都能追到来源（赔率行、向量块、条款、规则文件）。
3. **验证点显性**：每个 phase 有客观可查的验收标准（不是"看起来对"）。
4. **缺数据不编造**：没有就是没有。用占位 / 代理 / 降级，但绝不生成源里没有的数字。

## 3 段式方法论（完全不懂领域时如何靠 AI 理清）
不依赖你自己懂领域，而是**逼 AI 当领域分析师 + 架构师，并 PROVE 其说法**。

### A 侦察（Recon）—— 逼 AI 当领域分析师
让 AI 产出：
- 概念表（每个专业术语 + 一句人话类比 + 它在系统里的角色）
- 分层架构图（三层骨架初稿）
- 坑点清单（该领域常见的数据 / 口径 / 合规陷阱）
- 数据源清单（哪些数据可得、免费 / 付费、结构化 / 非结构化）
- 对每个说法要求给出"为什么可信"（出处 / 文献 / 已知成熟做法）

### B 拆解（Decompose）—— 逼 AI 列有序 phase
让 AI 产出带验收 Gate 的 phase 列表：
- 每个 phase：子目标 + 硬性约束 + 完成标准（可客观检查）
- 标注高风险 phase（易过拟合 / 易泄漏 / 数据难拿）
- 列出分支（某 phase 失败时的备选）
- **由此自然得出"总步数"**——用户最关心的"一共多少步"
- 已知终点时：把终点切成可控 phase，逐个验收（阶段闸门法）

### C 建造（Build）—— 逐 phase 交付
- 每次只给一个 phase 的子目标 + 约束 + 完成标准
- 客观验收（跑测试 / 对真实样本 / 看数值是否合理），不过关就回滚
- 收尾逼 AI 列"无法验证、只是推测"的项，写入交付说明

## 阶段闸门（Phase Gate）模板
每个 phase 结束前，必须能回答：
- [ ] 输出是什么（具体文件 / 数值）？
- [ ] 怎么客观验证它是对的？（测试 / 样本 / 对比）
- [ ] 缺 / 脏数据怎么处理的？有没有编造？
- [ ] 这一步的输出，下一步能否溯源到它？
不过关 → 不进下一 phase。

## 贯穿 5 问（每个 phase 都过一遍）
1. 如何**客观验证**这一步的结果？
2. **缺 / 脏数据**怎么办？（禁编造，要降级标注）
3. 数字能否**溯源**到来源？
4. 是否**过拟合 / 泄漏**（用未来数据训、全历史拟合称效果好）？
5. 交付里哪些**近似 / 降级 / 未验证**？

## 反模式红旗（出现即叫停）
- 🚩 AI 直接吐结论，没有确定性内核
- 🚩 单脚本全干（解析 + 算 + 解释混一起）
- 🚩 拿全历史拟合称"效果好"
- 🚩 缺数据就编个值填上
- 🚩 报告里出现源里没有的新数字
- 🚩 一次性甩整套架构试图"一步到位"
- 🚩 没有缺数据降级路径
- 🚩 随机模拟不固定种子、不可复现
- 🚩 LLM 既算又解释（角色没分开）

**结论**：17 个项目骨架同构，但只有 11 个真正落地了"确定性内核"；**6 个是薄壳**（LLM 抢硬计算或解释层空壳）。这 6 个恰恰是"外行靠 AI 容易写出来、但一碰就碎"的典型——本 Skill 的价值就是填这条沟：既教 GOOD 版怎么抄，也教 WEAK 版怎么纠正成 GOOD。

**源码带来的 6 个关键洞见**（已写进方法论）：
1. **确定性内核 ≠ 必须复杂 ML**：加权公式、纯算术、规则表、时间分块、数据集映射、查表——越轻越好，别一上来就上向量库。
2. **内核可委托给成熟外部系统**（green-deck→Anki；activity-timecard 甚至零 AI）。
3. **生成类护栏 = 源校验流水线**（green-deck 的 `src` 映回原文）。
4. **"计算/解释真分离"最简自检 = 内核文件 `grep` 不到任何 LLM import**。FitMate/CookHero/ai-travel-planner/AI-study-planner 都过不了这条。
5. **解释层缺失比内核错更隐蔽**（AI_Portfolio_Analyzer 空壳）——验收必须把"解释层真接上"列为 Gate。
6. **17 个里 6 个薄壳**：证明"外行靠 AI 写出来"与"写出 robust 项目"之间有方法论鸿沟，本 Skill 即填沟之用。

## 反模式纠正流程：把薄壳改造成有内核的项目
遇到 WEAK/ANTI 类项目（或你自己/AI 写出了"LLM 既算又解释"的版本），按此流程纠正：

1. **定位被 LLM 吃掉的硬计算**：问"这个成品里，哪些数字/决定是 LLM 直接编的？"（如 FitMate 的动作选择、ai-travel-planner 的排程、CookHero 的选菜）。
2. **逼出确定性替代**：和 AI 一起把那部分重写成"规则/公式/查表/算法"——
   - 匹配类 → 加权关键词/规则表（Resume-Matcher、activity-timecard）
   - 排程类 → 时间分块算法（study-planner）
   - 营养/财务类 → 公开公式 + 查表（LossWeightEasily、maybe）
   - 约束类 → 求解器（CookHero 应有）
3. **把 LLM 降级为解释层**：原 LLM 调用改为"消费已算好的内核结果，只生成文案/润色"。
4. **补解释层防空壳**：若原版解释层是空壳（如 AI_Portfolio_Analyzer），把内核指标拼成结构化输入喂给 LLM 写报告，列 Gate 验收"真接上"。
5. **自检**：内核文件 `grep` 无 LLM import；核心数字可手算复核；LLM 失败有降级。

## 复现模式：用本 Skill 一个一个做出来
用户最终要"把项目全复现出来"时，驱动方式：
1. 从 `references/reproduce.md` 取目标项目的**确定性内核规格 + 验收 Gate + 参考仓库路径**。
2. 走 A 侦察 → B 拆解 → C 建造：先逼 AI 把内核规格落成可运行代码（过 Gate），再接 AI 解释层。
3. WEAK 项目按"反模式纠正流程"复现（产出优于原版）；GOOD 项目照模板复现。
4. 每项目收尾过"复现通用验收清单"（内核零 LLM import / 数字可复核 / 缺数据降级 / 可溯源）。
5. 17 个项目覆盖 11 种范式，复完成时即掌握"任意陌生领域都能照此法落地"的能力。

## 交付清单（给用户的最终产物）
建造完成后，至少交付：
1. 三层骨架图（你的项目落在哪层、内核算什么）
2. phase 列表 + 各自 Gate 状态
3. 验证点说明（怎么证明对）
4. 诚实声明（降级 / 未验证 / 推测项）
5. 复现方式（种子 / 数据版本 / 依赖）
