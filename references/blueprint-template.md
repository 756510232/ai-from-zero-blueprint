# 固化蓝图模板（AI 主动套用，勿每次向用户重述）

> 本文件是 `ai-from-zero-blueprint` 的"通用零件库"。每个新项目都**先套用**这些模板，再填充
> 本领域的**具体内容**。AI 拿到用户目标后应**主动产出**，不向用户重复询问"这三层是什么 /
> 有哪些陷阱 / 需要什么数据"——这些是跨领域通用的，用户无需懂，也无需每次确认。
>
> 用户给的永远是"产品意图"（如"做一个张雪峰式高考志愿网页应用"），不是"复盘/复现某系统"
> 之类的元任务。AI 不得假设终点，终点由 brainstorming 阶段与用户共同确认（见 SKILL.md 第零步）。

---

## 1. 三层同构骨架（强制，每项目必落）

```
① 确定性 / 检索内核  → 产生客观结果（概率 / 排名 / 抽取 / 分数 / 调度 / 路由判定）
② 调度 / 路由层      → 决定调哪个内核、何时调、如何组合、把结果交给谁
③ AI 解释层          → 只把内核结果翻译成自然语言 / 生成内容，绝不参与计算或判定
```

- 每层必须明确写 **「负责什么 + 明确不负责什么」**。
- 若某层当前无 AI（如纯规则分类、纯算法排程），直接写 **「无 AI」**，这本身就是正确落点。
- **铁律自检**：内核代码文件 `grep` 不到任何 LLM / AI import。过不了这条 = 架构有根本问题。

---

## 2. 领域常见陷阱（6 类通用，先套类、再填本项目实例）

每个项目主动套这 6 类，逼 AI 给出「本项目里的具体实例 + 怎么防」：

- **T1 数据口径 / 量纲不一致**：别名归一化（队名 / 校名 / 商品名）、单位（美元 / 人民币、
  大卡 / 千焦）、年份 / 版本错配、跨源字段名不同。
- **T2 可复现性缺失**：随机模拟不固定种子；外部 API 漂移导致同输入不同输出；依赖版本未锁。
- **T3 过拟合 / 数据泄漏**：用未来数据训练；拿全历史拟合号称"效果好"；测试集泄漏进训练集。
- **T4 缺数据编造**：源里没有的数字被 LLM 编出来填进报告 / 界面。
- **T5 降级路径缺失**：某数据源 / 模型失败即整站 500，无 fallback，无占位。
- **T6 合规 / 隐私 / 版权**：训练数据权属、用户隐私、生成内容版权、领域监管（医疗 / 金融 /
  教育尤其敏感）。

## 3. 数据需求清单模板（主动填，每项目）

对每个所需数据，标注三栏：

| 数据 | 形态 | 可得性 | 降级映射（拿不到 / 脏了怎么办 + 对外如何诚实标注） |
|---|---|---|---|
| 例：历年录取分数线 | 结构化（DB/CSV） | 免费可得 | 缺失年份→标注"无数据"，不编 |
| 例：院校简介 | 非结构化（文档） | 免费可得 | 解析失败→跳过该项，标注"未收录" |

- 形态：结构化（DB / CSV）/ 非结构化（文档 / 网页 / 图片）
- 可得性：免费可得 / 付费 / 根本拿不到
- 降级映射：拿不到或脏了 → 代理值 / 占位 / 跳过，且对外如何诚实标注

---

## 4. 标准 phase 顺序（所有项目同构，主动套用，勿每次"切模块"）

固定 6 个 phase，**按序推进**；每个 phase 用下方「四要素」模板填充本项目内容。
（这套顺序是方法论沉淀的通用骨架，与具体领域无关——AI 直接套，不必让用户重新"定义模块"。）

- **P1 Schema 先行**：定义核心数据结构与字段（先定"数据长什么样"，不写逻辑）。
- **P2 最小确定性内核**：不接 AI、不接外部数据，用假数据 / 小样本先把核心计算跑通。
- **P3 接入真实数据**：替换假数据，处理口径 / 缺失 / 降级。
- **P4 接 AI 解释层**：只消费内核结果生成文案 / 回答，不参与计算。
- **P5 健壮性与诚实降级**：外部失败 fallback、缺数据标注、可复现（固定种子 / 锁版本）。
- **P6 验证与诚实声明**：对真实样本验收，收尾列「已验证 / 推测 / 降级 / 复现所需」。

### 每 phase 四要素（拆解阶段必须写清，建造阶段照此检测）

| 要素 | 含义 | 示例 |
|---|---|---|
| **约束目标** | 这一步要达成什么、不能做什么 | "禁止用未来数据""内核不得 import LLM" |
| **完成标准** | 客观可检查 | "跑通 10 条样本、数值落合理区间、测试通过" |
| **风险** | 易错点 | "易过拟合 / 数据泄漏 / 外部漂移 / 数据难拿" |
| **检测方法** | 怎么客观验证对了 | "对真实样本复核 / 手算对照 / 单元测试 / 对比已知答案" |

> 拆解阶段的核心交付物就是这张「每 phase 四要素表」。用户看它不是为了"知道步骤"，而是
> 为了明白**每一步的边界与验收口径**，这样建造阶段才能逐项检测，不会稀里糊涂"看起来对"。
