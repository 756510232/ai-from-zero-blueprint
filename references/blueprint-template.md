# 固化蓝图模板（AI 主动套用，勿每次向用户重述）

> 本文件是 `ai-from-zero-blueprint` 的"通用零件库"。每个新项目都**先套用**这些模板，再填充
> 本领域的**具体内容**。AI 拿到用户目标后应**主动产出**，不向用户重复询问"这三层是什么 /
> 有哪些陷阱 / 需要什么数据"——这些是跨领域通用的，用户无需懂，也无需每次确认。
>
> 用户给的永远是"产品意图"（如"做一个张雪峰式高考志愿网页应用"），不是"复盘/复现某系统"
> 之类的元任务。AI 不得假设终点，终点由 brainstorming 阶段与用户共同确认（见 SKILL.md 第零步）。

---

## 0. 护栏分级（先定档，再套强制动作）

模板 §4 里的"强制动作"**不是一刀切**。先按下面两维给项目定档，档位决定 P1 / P2 强制动作的松紧：

| 档位 | 判定（满足任一即升档） | 典型场景 |
|---|---|---|
| **S 小 / 低风险** | 单文件脚本、纯练习、无外部依赖、不涉及钱 / 健康 / 合规 / 对外发布 | 个人记账 demo、学习卡片 toy、内部玩具 |
| **M 中 / 常规** | 多文件、有外部数据 / 展示层、单人维护、非敏感领域 | 普通信息类应用、内部小工具 |
| **L 大 / 高风险** | 涉及钱 / 医疗 / 教育 / 法律建议、多人协作、长期维护、对外发布 | 志愿建议、投顾、合同审查、SaaS |

**分级套用规则**（P1 / P2 强制动作的松紧随档位变化）：

| 强制动作 | S 级 | M 级 | L 级 |
|---|---|---|---|
| P1 单一共享 schema 契约 | 轻量：代码顶部注释列字段即可 | **必做**：一份契约文件 | **必做** + 契约变更需评审 |
| P1 loader 断言表头 === 契约 | 省略（单源、字段稳定） | **必做** | **必做** |
| P1 读前剥 `#` 注释行 / 空行 | 建议 | 建议 | **必做**（防注释吞表头） |
| P2 锁方向 / 符号约定单测 | 轻量：注释写死方向 + 手算对照 3 样本 | **必做**：≥3 边界用例单测文件 | **必做** + CI 门禁（不过不合并） |
| P5 降级态有集成测试 | 省略 | 建议 | **必做** |

> **原则**：S 级**不**因为"模板要求"而被迫写工程化脚手架——它要的是"方向约定写死、手算对照"，不是 pytest 框架。
> 但凡涉及钱 / 健康 / 教育 / 合规 / 对外发布（L 级），单测与契约就是**硬门槛，没有商量**。
> 定档由 AI 在 A 侦察阶段给出并请用户确认一行即可（不重开 brainstorming）。

## 1. 三层同构骨架（强制，每项目必落）

```
① 确定性 / 检索内核  → 产生客观结果（概率 / 排名 / 抽取 / 分数 / 调度 / 路由判定）
② 调度 / 路由层      → 决定调哪个内核、何时调、如何组合、把结果交给谁
③ AI 解释层          → 只把内核结果翻译成自然语言 / 生成内容，绝不参与计算或判定
```

- 每层必须明确写 **「负责什么 + 明确不负责什么」**。
- 若某层当前无 AI（如纯规则分类、纯算法排程），直接写 **「无 AI」**，这本身就是正确落点。
- **铁律自检**：内核代码文件 `grep` 不到任何 LLM / AI import。过不了这条 = 架构有根本问题。

## 2. 领域常见陷阱（6 类通用，先套类、再填本项目实例）

每个项目主动套这 6 类，逼 AI 给出「本项目里的具体实例 + 怎么防」：

- **T1 数据口径 / 量纲不一致**：别名归一化（队名 / 校名 / 商品名）、单位（美元 / 人民币、
  大卡 / 千焦）、年份 / 版本错配、跨源字段名不同。
  **额外（跨口径估算）**：用户给位次、库只给分数（或反之）时的硬算会误导——
  必须标"⚠近似"并降置信度，不得当精确结论。
- **T2 可复现性缺失**：随机模拟不固定种子；外部 API 漂移导致同输入不同输出；依赖版本未锁。
- **T3 过拟合 / 数据泄漏**：用未来数据训练；拿全历史拟合号称"效果好"；测试集泄漏进训练集。
- **T4 缺数据编造**：源里没有的数字被 LLM 编出来填进报告 / 界面。
- **T5 降级缺失 / 硬约束被静默跳过**：① 某数据源 / 模型失败即整站 500，无 fallback，无占位；
  ② **硬约束（资格 / 选科 / 门槛）必须在内核显式 exclude，绝不可写成 `if 不满足: pass` 静默放过**；
  ③ **降级必须是显式、独立的状态（枚举 / 独立标签），不得把"缺数据行"算成正常结论再补一句注（伪装成推荐）**。
- **T6 合规 / 隐私 / 版权**：训练数据权属、用户隐私、生成内容版权、领域监管（医疗 / 金融 /
  教育尤其敏感）。

## 3. 数据需求清单模板（主动填，每项目）

对每个所需数据，标注三栏：

| 数据 | 形态 | 可得性 | 降级映射（拿不到 / 脏了怎么办 + 对外如何诚实标注） |
|---|---|---|---|
| 例：历年录取分数线 | 结构化（DB/CSV） | 免费可得 | 缺失年份→标注"无数据"，不编 |
| 例：院校简介 | 非结构化（文档） | 免费可得 | 解析失败→跳过该项，标注"未收录" |

- 形态：结构化（DB / CSV）/ 非结构化（文档 / 网页 / 图片）
- 可得性：免费可得 / 付费 / 根本拿不到
- 降级映射：拿不到或脏了 → 代理值 / 占位 / 跳过，且对外如何诚实标注

## 4. 标准 phase 顺序（所有项目同构，主动套用，勿每次"切模块"）

固定 6 个 phase，**按序推进，不得跳过**。AI 主动套用，不必让用户重新"定义模块"。

**每个 phase 的前置条件不通过 → 不得进入当前 phase**。任何跳过行为应被视为 T5（硬约束被静默跳过）。

下面是每个 phase 的完整四要素。拆解阶段必须写清，建造阶段照此逐项检测。

### P1 — Schema 先行（单一共享契约）

| 要素 | 内容 |
|---|---|
| **前置条件** | 无（起始 phase） |
| **约束目标** | 先定"数据长什么样" + "输出长什么样（含降级态枚举）"，不写任何逻辑代码。输出结构必须包含 `NO_DATA / UNKNOWN / 不符合` 等降级态枚举，禁用散字符串。 |
| **完成标准** | ① 一份独立共享 schema 契约文件（输入结构 + 输出结构，降级态是枚举非字符串）；② 数据文件（如 CSV）表头字段 === 契约字段（loader 必须有断言，不符立即报错，禁止跨文件字段名漂移）；③ CSV/文本的 loader 先剥 `#` 注释行与空行，避免注释行被当表头导致字段错位 |
| **风险** | 最容易被省略 — AI 觉得"契约写代码注释里就行"或"字段名差不多就行"。跳 P1 会导致后续 phase 字段漂移、loader 表头错位 |
| **检测方法** | ① `grep` 契约文件存在且字段定义完整；② 加载一份 dummy 数据，loader 断言必须通过；③ 故意给错字段名，确认 loader 报错而非沉默 |

### P2 — 最小确定性内核

| 要素 | 内容 |
|---|---|
| **前置条件** | P1 ✅ |
| **约束目标** | 不接 AI、不接外部数据，用假数据 / 小样本先把核心计算跑通。内核文件**零 LLM / AI import**（铁律）。先选最轻的内核形态，再写最小版。 |
| **完成标准** | ① 内核能从假数据算出正确结果（方向/符号逻辑与领域常识一致）；② 单测（M/L 级）或注释+手算对照 3 样本（S 级）锁死方向约定，单测能在接真数据前抓出量纲反转类 bug；③ `grep -rilE "llm|openai|anthropic|chatgpt" kernel.*` 零命中 |
| **风险** | 方向约定与领域常识相反（如"用户占优→冲"）；量纲反转（越大越好/越小越好搞反）；跳过内核直接用 LLM 编答案（纯薄壳） |
| **检测方法** | ① 单测断言业务语义结果（如"位次差为正→稳"，不是只锁数值）；② 手算 3 个边界样本对照单测输出；③ `grep` 内核文件无 LLM import；④ 覆盖边界：空输入、异常值、极端边界 |

**内核形态选项库（先选最轻的形态，再写最小版）**——v2 证据库归纳出约 14 种可复用内核形态：
- 检索类：向量检索 / 混合检索（Haversine 地理 + 元数据过滤）
- 规则类：加权公式 / 规则表 / DSL 引擎（AST 合一匹配）/ 立法规则图 / 语法规则
- 算法类：Dijkstra / 0-1 背包 DP / 符号计算 / 乐理计算
- 最优化类：LP·MILP / 回测引擎 / EOQ 库存优化
- 数据类：查表 / 统一数据接口+多源适配 / 强类型领域模型（实体+关系+校验）
- 神经推理类：本地模型推理（CTranslate2，确定性、非 LLM 自由生成）
> 原则：**越轻越好**，别一上来就上向量库 / ML；加速（CH/MLD/向量索引/缓存）是后置优化，先让朴素版（Dijkstra / 单 pattern 匹配 / 直接查表）跑通。

### P3 — 接入真实数据

| 要素 | 内容 |
|---|---|
| **前置条件** | P2 ✅ |
| **约束目标** | 用真实、可溯源的数据替换 P2 的假数据。数据来源必须明确标注，非伪造/模拟/LLM 生成。处理口径/缺失/降级。 |
| **完成标准** | ① loader 能从真实数据源读入并解析；② 数据文件头标注 `SOURCE: <URL/来源>`；③ 缺数据行标 `NO_DATA` 不编造值；④ 验证数据字段与 P1 契约一致；⑤ 真实部署场景的数据不可得时 → 项目降级为 DRAFT |
| **风险** | ⚠️ **最易被跳过** — AI 觉得"数据不好下载"就自作主张继续用假数据做 P4/P5，花大量 token 编造在虚假数据上的"结果"，无意义 |
| **检测方法** | ① 随机抽 3 行反向查证（数据能溯源到官方发布源）；② `grep` 数据文件确认无 `FakeData`/`合成`/`模拟` 关键词；③ 故意断掉数据源，确认 loader 输出 `NO_DATA` 而非沉默或编值；④ 数据文件的每一行都能追到具体来源标注 |

> **P3 是基础 phase，不可跳过**。如果真实数据确实不可得 → 项目状态自动降级为 **DRAFT / DEMO-ONLY**。P4/P5/P6 仍可执行，但每层输出必须带 `⚠️ 数据源非真实 → DEMO 模式` 标记。✅ **允许用样本数据做 P4/P5 的开发和测试**，但前提是：
> - 样本数据文件头标注 `NON_REAL — DEMO ONLY`
> - P6 诚实声明首要条目列明：`跳过 phase: P3（原因：真实数据不可得），项目状态：DRAFT`
> - ❌ **不得以"已通过验证"结论输出**
> - ❌ **"用假数据跑通 P4/P5 然后在末尾补一句'数据是假的'"** 属于 T5（硬约束被静默跳过），不是诚实降级

### P4 — 接 AI 解释层

| 要素 | 内容 |
|---|---|
| **前置条件** | P3 ✅ |
| **约束目标** | AI 解释层**只消费内核结果**生成文案/回答，**不参与任何计算**。解释层必须带确定性模板兜底（拔掉 LLM 仍能输出结构化解释，不崩、不编造）。不得用 LLM 做内核该做的事（如分档、算分）。 |
| **完成标准** | ① 解释层输入 = P2 内核的输出（三档/分数/指标等），不与数据源直接交互；② LLM 调用是**可选**的，去掉后项目仍能完整运行；③ 解释里出现的每个数字都能在数据源找到对应行 |
| **风险** | LLM 编造数据源不存在的数字；解释层代行内核职能（如让 LLM 做分档）；LLM 失败时无兜底 |
| **检测方法** | ① 关掉 LLM，确认项目仍能运行并输出结构化解释；② 对解释中每个数字，追查它来自数据源的哪一行；③ 模拟 LLM 超时/报错，确认回退到模板文案且不报 500 |

### P5 — 健壮性与诚实降级

| 要素 | 内容 |
|---|---|
| **前置条件** | P4 ✅ |
| **约束目标** | 所有外部失败（DB/API/LLM/文件系统）都有 fallback 路径；缺数据显式标注不编造；运行可复现（固定种子/锁依赖版本）。 |
| **完成标准** | ① DB 不可用 → 降级为样本模式并显式标注；② LLM 不可用 → 回退模板文案；③ 运行日志记录每次降级事件；④ 固定随机种子和依赖版本 |
| **风险** | 最容易被"跑通了，降级以后再说"跳过。线上断网/数据源挂掉直接 500 |
| **检测方法** | ① 模拟断网/断 DB/断 LLM，确认不崩不 500；② 检查日志是否有降级事件记录；③ 用同一份输入 + 固定种子跑两次，确认输出完全一致 |

### P6 — 验证与诚实声明

| 要素 | 内容 |
|---|---|
| **前置条件** | P1–P5 全部 ✅ |
| **约束目标** | 对真实样本做最终验收。收尾输出诚实的完整状态声明。 |
| **完成标准** | ① 跑通多样本验证（含正常/边界/降级场景）；② 输出 **phase 全景表**（6 行，每行状态：已完成 / 跳过 / 降级）；③ 诚实声明覆盖：已验证 / 推测 / 降级 / 复现所需（种子/数据版本/依赖） |
| **风险** | 编造测试结果声称"全部通过"；不记录降级和推测项 |
| **检测方法** | ① 对比 P2 单测、P3 真实数据、P4 解释层的输出是否可串成全链路；② phase 全景表中存在"跳过"→ 整项目状态降为 DRAFT；③ 按诚实声明中的种子/版本/依赖重跑一次，确认可复现 |

> 拆解阶段的核心交付物就是这张「每 phase 四要素表」。用户看它不是为了"知道步骤"，而是
> 为了明白**每一步的边界与验收口径**，这样建造阶段才能逐项检测，不会稀里糊涂"看起来对"。
